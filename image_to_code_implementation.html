image_to_code.txt Phase 6.3

Implementation outline for the dedicated image-to-code prompt template and its integration across the LatticeLabs generation pipeline.

1. Problem Statement

The implementation outline specified five prompt template files under cadling/generation/codegen/prompts/. Four were implemented; image_to_code.txt was missing. The existing text_to_code.txt contained a generic {IMAGE_SECTION} placeholder that injected a minimal 6-line image preamble when an image was provided — but it lacked the structured analysis guidance an LLM needs to reliably reverse-engineer 3D geometry from a 2D image.

The OpenSCADGenerator had no image support at all — its generate() method accepted only description: str and always stripped the {IMAGE_SECTION} placeholder.

2. What Was Created

cadling/generation/codegen/prompts/image_to_code.txt new

A dedicated prompt template for image-primary CAD code generation. 175 lines containing six structured sections that guide the LLM through a rigorous image-to-geometry pipeline.

Template Structure

SectionPlaceholderPurpose
Task — Sets the objective: generate executable CAD code from a reference image that produces a watertight solid.
Reference Image {IMAGE_DATA} Receives the base64 data URI with MIME type. Tells the LLM to analyze geometry, proportions, and features.
Supplementary Text {DESCRIPTION} Optional text description. When present, its dimensions override image-inferred values. Set to "(none provided)" when absent.
Image Analysis Instructions — Four-step analysis framework the LLM must follow before writing code:
Step 1 — Identify overall form (shape class, aspect ratio, symmetry)
Step 2 — Decompose into features (additive, subtractive, transition, pattern, surface)
Step 3 — Estimate dimensions in mm (proportional reasoning, standard hardware mapping)
Step 4 — Plan modeling sequence (base → secondary → cuts → fillets)
Multi-View Guide — Interpretation rules for isometric, orthographic front/side/top, section views, and detail views. Cross-reference conflict resolution (prefer orthographic over perspective).
Image Type Handling — Adaptation strategies for 5 image source types: engineering drawings (extract annotated dims), CAD screenshots (use shading cues), photographs (account for perspective distortion), hand sketches (focus on topology), exploded views (generate single component).
Code Requirements — Parametric variables at top, commented modeling steps mapped to image features, occluded geometry handling with INFERRED/ASSUMED/UNCERTAIN annotations.
Output Format — Fenced code block (python for CadQuery, openscad for OpenSCAD).

Key Design Decisions

Why a separate template instead of extending text_to_code.txt? — The image-primary workflow requires fundamentally different cognitive scaffolding. A text-primary prompt says "here's what to build"; an image-primary prompt says "figure out what to build from this picture." The analysis steps (form ID → feature decomposition → dimension estimation → sequence planning) are specific to visual reasoning and would bloat the text-only path unnecessarily.

6. Usage Examples

CLI: Image-only generation (uses image_to_code.txt)

cadling generate --from-image photo_of_bracket.png --output bracket.step

No --from-text flag → CLI generates auto-description → detected as placeholder → routes to image_to_code.txt template with full analysis instructions.

CLI: Image + text generation (uses text_to_code.txt with image section)

cadling generate \
  --from-text "A flanged bearing housing, 120mm OD, 4 M8 bolt holes" \
  --from-image bearing_sketch.png \
  --output housing.step

Substantive text description → routes to text_to_code.txt with the image injected into {IMAGE_SECTION}. Text dimensions take precedence; image informs shape/proportions.

Python API: Image-primary with CadQuery backend

from cadling.generation.codegen.cadquery_generator import CadQueryGenerator

gen = CadQueryGenerator(model_name="claude-3-opus", api_provider="anthropic")
script = gen.generate(
    description="",  # empty → triggers image_to_code.txt
    reference_image_path="renders/housing_iso.png",
)

Python API: Image-primary with OpenSCAD backend

from cadling.generation.codegen.openscad_generator import OpenSCADGenerator

gen = OpenSCADGenerator(model_name="gpt-4o", api_provider="openai")
script = gen.generate(
    description="",  # empty → triggers image_to_code.txt
    reference_image_path="renders/housing_iso.png",
)

Full pipeline: Image → validated STEP

from cadling.generation.pipeline import GenerationPipeline
from cadling.datamodel.generation import GenerationRequest, GenerationConfig, GenerationBackend

config = GenerationConfig(
    backend=GenerationBackend.CODEGEN_CADQUERY,
    validate_output=True,
    max_retries=3,
)
pipeline = GenerationPipeline(config)

request = GenerationRequest(
    image_path="photo_of_part.jpg",
    # text_prompt omitted → pipeline auto-generates placeholder → image_to_code.txt
)
results = pipeline.generate(request)

for r in results:
    print(f"Valid: {r.valid}, Path: {r.output_path}")

8. Design Rationale

Why structured analysis steps?

LLMs generating CAD from images without structured guidance tend to produce oversimplified geometry — they'll see a bracket and output a box with two holes, missing ribs, fillets, mounting ears, and slot features. The four-step analysis framework (form → features → dimensions → sequence) forces the model to enumerate features before writing code, dramatically improving completeness. This mirrors how the Text2CAD paper structures its "expert-level" annotations.

Why image type handling?

A photograph of a machined part, an engineering drawing, and a hand sketch all contain geometric information but require fundamentally different interpretation strategies. Perspective distortion in photos, annotated dimensions in drawings, and topological-but-not-dimensional accuracy in sketches each need explicit handling instructions. Without this, LLMs apply photograph-appropriate reasoning to all image types.

Why confidence annotations?

Occluded geometry is the primary failure mode in image-to-CAD. A single isometric view of a part hides roughly 50% of the geometry. The INFERRED / ASSUMED / UNCERTAIN comment convention makes hidden assumptions explicit, enabling downstream humans (or validation loops) to identify which geometric choices to verify.

Why standard hole size mapping?

The most common image-to-CAD dimension error is hole sizing. Providing the ISO metric clearance hole table (M3=3.2mm through M10=10.5mm) in the template gives the LLM a concrete lookup table for the most frequently occurring features, reducing dimension estimation errors for the highest-value geometry.
